# Учебнаяа практика

Десктопное приложение для управления сетевыми хранилищами

## 1. Выбор темы проекта и обоснование модели проектирования

В качестве темы проекта было дано задание создание десктопного приложения для управления сетевыми хранилищами. Данная тема с работой с сетевыми дисками и файловыми серверами является распространённой задачей, особенно в сфере IT. При этом существующие решения либо работают через браузер, либо имеют ограниченный функционал, поэтому разработка отдельного десктопного приложения выглядит актуальной и полезной.

В рамках проекта планируется реализовать программу, которая позволит подключаться к сетевым хранилищам, просматривать файлы и выполнять основные операции с ними. Это делает проект практико-ориентированным и полезным для дальнейшего применения.

При выборе модели разработки рассматривались два подхода: Waterfall и Agile. Модель Waterfall предполагает строгую последовательность этапов, где каждый следующий начинается только после завершения предыдущего. Такой подход удобен, если все требования заранее известны и не изменяются.

Однако в данном проекте требования могут уточняться по мере разработки, а функциональность может постепенно расширяться. Поэтому более подходящей является модель Agile, которая позволяет работать итерациями, постепенно добавляя новые возможности и внося изменения при необходимости.

Таким образом, для разработки данного проекта была выбрана методология Agile, так как она лучше подходит для гибкой и поэтапной разработки приложения.

## 2. Проведение предпроектных исследований: анализ существующих аналогов, определение стейкхолдеров

Перед началом разработки был проведён анализ существующих решений для работы с сетевыми хранилищами. Это позволило понять, какие функции уже реализованы в других программах и чего в них не хватает.

Были рассмотрены такие решения, как TrueNAS и Synology DSM. Они предоставляют широкий набор возможностей для управления хранилищами, однако работают через веб-интерфейс и часто зависят от конкретного оборудования. Также был рассмотрен стандартный проводник Windows, который позволяет подключаться к сетевым папкам, но имеет только базовый функционал и не подходит для полноценного администрирования.

В результате анализа можно сделать вывод, что существующие решения либо слишком узкоспециализированы, либо не обеспечивают удобного и универсального способа управления. Это подтверждает актуальность разработки собственного десктопного приложения.

Также на данном этапе были определены основные стейкхолдеры проекта. В первую очередь это конечные пользователи — системные администраторы и продвинутые пользователи, которым необходимо работать с сетевыми хранилищами. Кроме того, к стейкхолдерам относится разработчик, так как он отвечает за реализацию и поддержку системы. В некоторых случаях стейкхолдером может выступать организация, в которой будет использоваться данное приложение.

Таким образом, проведённый анализ позволил лучше понять требования к системе и определить круг заинтересованных лиц, что важно для дальнейшей разработки.

## 3. Создание репозитория в Git (структура docs/, src/, tests/)

Для разработки проекта был создан репозиторий в системе контроля версий Git. Это необходимо для удобного хранения кода, отслеживания изменений и возможности дальнейшего развития проекта.

В репозитории была сразу организована базовая структура папок. Папка src/ предназначена для исходного кода приложения. В ней будет находиться вся логика программы. Папка docs/ используется для хранения документации, например описания проекта, требований и других текстовых материалов. Папка tests/ предназначена для тестов, которые будут проверять корректность работы программы.

Также в корне репозитория был создан файл README.md с кратким описанием проекта, а в дальнейшем планируется добавить файл с требованиями и другой документацией.

Такая структура позволяет удобно организовать проект и упрощает работу с ним как на этапе разработки, так и в дальнейшем.

## 4. Сбор требований: составление списка функциональных и нефункциональных требований (FURPS+)

На данном этапе были определены основные требования к разрабатываемому приложению. В первую очередь были выделены функциональные требования, то есть то, что программа должна уметь делать. Приложение должно поддерживать подключение к сетевым хранилищам по таким протоколам, как SMB, FTP и SFTP. Пользователь должен иметь возможность просматривать структуру файлов и папок, а также выполнять основные операции: загружать и скачивать файлы, удалять их и переименовывать. Кроме этого, система должна позволять работать сразу с несколькими подключениями и отображать информацию о состоянии хранилища, например его доступность и объём свободного места.

Помимо этого были рассмотрены нефункциональные требования по модели FURPS+. Интерфейс программы должен быть простым и понятным, чтобы пользователь мог быстро разобраться в работе приложения. Важно обеспечить стабильную работу программы и корректную обработку ошибок, особенно при проблемах с подключением. Также система должна работать достаточно быстро, без заметных задержек при выполнении операций с файлами.

Отдельное внимание уделяется поддерживаемости и расширяемости программы — структура проекта должна позволять легко вносить изменения и добавлять новый функционал. Также важным требованием является безопасность: данные пользователя должны передаваться с использованием защищённых протоколов, а доступ к хранилищам должен осуществляться только после авторизации.

Таким образом, на данном этапе был сформирован набор требований, который будет использоваться в дальнейшем при проектировании и разработке приложения.
